# Data Model Suggestion 2: Event-Sourced / Audit-First (CQRS)

> Project: Succession Planning Tool · Created: 2026-05-12

## Philosophy

This model treats every succession-related action as an immutable domain event stored in an append-only event store. The event store is the single source of truth. Materialised read models (projections) are rebuilt from events to serve different query patterns: the 9-box grid view, bench strength dashboards, ISO 30433 metrics, and board-ready reports each read from purpose-built projections optimised for their access pattern.

This approach is ideal for succession planning because the domain is inherently temporal and audit-sensitive. Regulators care about *when* a readiness rating changed, *who* changed it, and *why*. GDPR Article 22 requires documenting the decision chain for AI-influenced employment decisions. ISO 30414/30433 metrics need to be computable for any historical period. Event sourcing provides all of this natively — every state transition is preserved, and any point-in-time view can be reconstructed by replaying events up to that timestamp.

The CQRS split also enables independent scaling: the write side (event ingestion) handles calibration sessions, nominations, and AI scoring bursts, while the read side (projections) serves dashboards and reports without contention. This mirrors the architecture described by Microsoft Azure's CQRS pattern and is used in financial services and healthcare for similar compliance-driven requirements.

**Best for:** Organisations requiring full temporal auditability, GDPR Article 22 compliance chains, and the ability to answer "what was true on date X?" for any succession data point.

**Trade-offs:**
- Pro: Complete, immutable audit trail with zero additional effort — every change is an event
- Pro: Any historical state can be reconstructed by replaying events to a point in time
- Pro: GDPR Article 22 decision chain is inherent — the sequence of AI recommendation, human review, and action is captured as events
- Pro: ISO 30433 metrics can be computed for any arbitrary historical period, not just pre-computed snapshots
- Pro: Read models can be optimised independently for each consumer (dashboard, report, API)
- Con: Higher implementation complexity — requires event store, projection engine, and eventual consistency handling
- Con: Read models are eventually consistent; calibration session participants may see stale data briefly
- Con: Event schema evolution requires careful versioning (upcasting old events to new formats)
- Con: Debugging requires replaying event streams rather than inspecting table rows
- Con: Team must understand event sourcing patterns; steeper learning curve for contributors

## Standards Alignment

| Standard | How It's Used |
|----------|---------------|
| ISO/TS 30433:2021 | Metrics are computed by projecting events (NominationCreated, ReadinessChanged, PromotionCompleted) over any time window — no pre-computed snapshots needed |
| ISO 30414:2025 | Human capital disclosure metrics are generated by running projection queries against the event store for the reporting period |
| GDPR Article 22 | The event chain AIRecommendationGenerated -> HumanReviewCompleted -> NominationConfirmed provides the legally required decision audit trail |
| SHRM Framework | Readiness tier transitions are captured as ReadinessChanged events with before/after values |
| ESCO Classification | SkillAssessed and SkillInferred events reference ESCO concept URIs |
| HR Open Standards | Event payloads use HR-JSON compatible field names for export compatibility |

## Event Store

```sql
-- The append-only event store — single source of truth
CREATE TABLE domain_event (
    event_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    stream_id UUID NOT NULL, -- aggregate root ID (e.g., succession_plan_id, employee_id)
    stream_type VARCHAR(50) NOT NULL, -- 'SuccessionPlan', 'Employee', 'CalibrationSession', 'TalentPool'
    event_type VARCHAR(100) NOT NULL, -- 'NominationCreated', 'ReadinessChanged', 'CalibrationCompleted', etc.
    event_version INTEGER NOT NULL, -- schema version of this event type for upcasting
    payload JSONB NOT NULL, -- event data
    metadata JSONB NOT NULL DEFAULT '{}', -- actor_id, ip_address, correlation_id, causation_id
    sequence_number BIGINT NOT NULL, -- global ordering
    occurred_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    -- Optimistic concurrency: stream_id + stream_version must be unique
    stream_version INTEGER NOT NULL,
    UNIQUE (stream_id, stream_version)
);

-- Global ordering index for projections that need total order
CREATE INDEX idx_event_sequence ON domain_event(sequence_number);
-- Stream replay
CREATE INDEX idx_event_stream ON domain_event(stream_id, stream_version);
-- Event type filtering for specific projections
CREATE INDEX idx_event_type ON domain_event(event_type);
-- Time-based queries
CREATE INDEX idx_event_occurred ON domain_event(occurred_at);

-- Sequence generator for global ordering
CREATE SEQUENCE domain_event_sequence_seq;
```

## Read Model Projections

```sql
-- Projection: Current succession plan state (rebuilt from events)
CREATE TABLE projection_succession_plan (
    plan_id UUID PRIMARY KEY,
    position_id UUID NOT NULL,
    position_title VARCHAR(255) NOT NULL,
    owner_id UUID NOT NULL,
    status VARCHAR(20) NOT NULL,
    total_successors INTEGER NOT NULL DEFAULT 0,
    ready_now_count INTEGER NOT NULL DEFAULT 0,
    ready_1_to_2_count INTEGER NOT NULL DEFAULT 0,
    ready_3_to_5_count INTEGER NOT NULL DEFAULT 0,
    last_reviewed_at TIMESTAMPTZ,
    last_event_sequence BIGINT NOT NULL, -- watermark for incremental rebuild
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_proj_plan_position ON projection_succession_plan(position_id);

-- Projection: Current nominations (denormalized for dashboard queries)
CREATE TABLE projection_nomination (
    nomination_id UUID PRIMARY KEY,
    plan_id UUID NOT NULL,
    position_id UUID NOT NULL,
    position_title VARCHAR(255) NOT NULL,
    employee_id UUID NOT NULL,
    employee_name VARCHAR(200) NOT NULL,
    readiness_tier VARCHAR(20) NOT NULL,
    rank_order INTEGER,
    nomination_status VARCHAR(20) NOT NULL,
    risk_of_loss VARCHAR(20),
    impact_of_loss VARCHAR(20),
    nominated_by_name VARCHAR(200),
    nominated_at TIMESTAMPTZ NOT NULL,
    last_event_sequence BIGINT NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_proj_nom_plan ON projection_nomination(plan_id);
CREATE INDEX idx_proj_nom_employee ON projection_nomination(employee_id);
CREATE INDEX idx_proj_nom_readiness ON projection_nomination(readiness_tier);

-- Projection: Current 9-box positions (latest assessment per employee)
CREATE TABLE projection_nine_box (
    employee_id UUID PRIMARY KEY,
    employee_name VARCHAR(200) NOT NULL,
    position_id UUID,
    position_title VARCHAR(255),
    organisation_unit_id UUID,
    performance_rating INTEGER NOT NULL,
    potential_rating INTEGER NOT NULL,
    nine_box_position INTEGER NOT NULL,
    assessment_source VARCHAR(30) NOT NULL,
    assessed_at TIMESTAMPTZ NOT NULL,
    last_event_sequence BIGINT NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_proj_nine_box_pos ON projection_nine_box(nine_box_position);
CREATE INDEX idx_proj_nine_box_org ON projection_nine_box(organisation_unit_id);

-- Projection: Bench strength summary (one row per key position)
CREATE TABLE projection_bench_strength (
    position_id UUID PRIMARY KEY,
    position_title VARCHAR(255) NOT NULL,
    organisation_unit_id UUID,
    level VARCHAR(50),
    is_key_position BOOLEAN NOT NULL DEFAULT true,
    incumbent_id UUID,
    incumbent_name VARCHAR(200),
    total_successors INTEGER NOT NULL DEFAULT 0,
    ready_now_count INTEGER NOT NULL DEFAULT 0,
    has_single_point_risk BOOLEAN NOT NULL DEFAULT false, -- true if total_successors <= 1
    risk_score NUMERIC(5,2), -- composite risk based on successor count, flight risk, readiness
    last_event_sequence BIGINT NOT NULL,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_proj_bench_risk ON projection_bench_strength(risk_score DESC);
CREATE INDEX idx_proj_bench_org ON projection_bench_strength(organisation_unit_id);

-- Projection: AI recommendation status (GDPR Article 22 tracking)
CREATE TABLE projection_ai_recommendation (
    recommendation_id UUID PRIMARY KEY,

    employee_id UUID NOT NULL,
    employee_name VARCHAR(200) NOT NULL,

    reviewer_name VARCHAR(200),
    reviewed_at TIMESTAMPTZ,
    algorithm_version VARCHAR(50),
```

## Example Queries

### Reconstruct succession plan state at a specific date

```sql
-- What was the bench strength for VP Finance on 2026-01-15?
SELECT
    payload->>'employee_id' AS successor_id,
    payload->>'readiness_tier' AS readiness_tier,
    payload->>'rank_order' AS rank_order
FROM domain_event
WHERE stream_type = 'SuccessionPlan'
  AND stream_id = (
      SELECT event_id FROM domain_event
      WHERE event_type = 'SuccessionPlanCreated'
        AND payload->>'position_id' = '<vp-finance-position-uuid>'
      LIMIT 1
  )
  AND event_type IN ('NominationCreated', 'ReadinessChanged', 'NominationWithdrawn')
  AND occurred_at <= '2026-01-15T23:59:59Z'
ORDER BY occurred_at;
```

### GDPR Article 22 decision chain for a specific recommendation

```sql
-- Full decision chain: AI recommendation -> human review -> action
SELECT
    event_type,
    payload,
    metadata->>'actor_id' AS actor,
    occurred_at
FROM domain_event
WHERE payload->>'recommendation_id' = '<recommendation-uuid>'
ORDER BY occurred_at;
-- Returns: AIRecommendationGenerated, HumanReviewCompleted, NominationCreated (or NominationRejected)
```

### Compute ISO 30433 metrics for any historical period

```sql
-- Successor coverage rate for Q1 2026
WITH key_positions AS (
    SELECT id FROM position WHERE is_key_position = true
),
positions_with_successors AS (
    SELECT DISTINCT payload->>'position_id' AS position_id
    FROM domain_event
    WHERE event_type = 'NominationCreated'
      AND occurred_at <= '2026-03-31T23:59:59Z'
      AND NOT EXISTS (
          SELECT 1 FROM domain_event d2
          WHERE d2.event_type = 'NominationWithdrawn'
            AND d2.payload->>'nomination_id' = domain_event.payload->>'nomination_id'
            AND d2.occurred_at <= '2026-03-31T23:59:59Z'
      )
)
SELECT
    (SELECT COUNT(*) FROM positions_with_successors WHERE position_id IN (SELECT id::text FROM key_positions))::NUMERIC
    /
    NULLIF((SELECT COUNT(*) FROM key_positions), 0)
    AS successor_coverage_rate;
```

## Key Design Decisions

1. **Single event table with JSONB payloads** — rather than one table per event type, all events share a single `domain_event` table with typed JSONB payloads. This simplifies event replay, subscription, and projection building while maintaining queryability via JSONB indexes.

2. **Global sequence number for total ordering** — the `sequence_number` column provides a total order across all streams, enabling projections that need to process events in causal order (e.g., bench strength depends on both nominations and departures).

3. **Optimistic concurrency via stream_version** — the UNIQUE constraint on `(stream_id, stream_version)` prevents concurrent writes to the same aggregate, protecting against calibration session race conditions.

4. **Projections are disposable and rebuildable** — every projection table can be dropped and rebuilt from the event store. This enables schema changes to read models without data migration — just rebuild the projection.

5. **Reference data stays relational** — organisation structure, positions, employees, and skills are not event-sourced because they originate from the HRIS and are updated via sync, not domain events. This avoids over-engineering the HRIS import pipeline.

6. **GDPR Article 22 compliance is structural** — the event chain `AIRecommendationGenerated -> HumanReviewCompleted -> NominationCreated` is an inherent property of the event sequence. There is no way to action a recommendation without the human review event existing in the stream.

7. **ISO 30433 metrics computed on-demand** — rather than storing pre-computed snapshots (as in the normalized model), metrics are computed by projecting events over any requested time window. This supports arbitrary reporting periods without data duplication.

8. **Event versioning for schema evolution** — the `event_version` column on each event enables upcasting (transforming old event formats to new ones during replay) without modifying stored events.

9. **No separate audit_log table** — the event store IS the audit log. Every state change is an event with actor metadata, eliminating the need for a parallel audit mechanism.
